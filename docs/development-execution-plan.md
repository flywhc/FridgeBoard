# FridgeBoard 分会话开发执行计划

状态：P0 已完成，P1 待启动  
更新日期：2026-07-19  
适用范围：从架构设计到上线前验收；不在本文预先锁定技术栈或部署平台。

## 1. 使用方式

本计划将开发拆为可在独立 Codex 会话中完成的任务包。每个会话只领取一个任务包；完成后更新 [进度看板](progress-tracker.md)，并在下一会话附上交接摘要。

每个任务包启动时都应提供四项信息：

1. **目标**：要交付的可观察结果。
2. **上下文**：相关文档、UI 草稿、已有代码和前置任务结果。
3. **约束**：不得改变的产品规则、兼容性和安全边界。
4. **完成条件**：要通过的测试、检查和人工验收点。

不要把“写代码”“测试”“评审”和“部署”混为同一个未定义的大任务。复杂或不确定的工作先使用 Plan 模式形成实施计划；行为改变必须有对应自动化测试或明确的手工验收记录。

### 1.1 设计基线

- 功能、交互和可行性基线：[功能设计与可行性分析](functional-design-and-feasibility.md)。
- 最终 UI 草稿 ID 索引：[功能设计与可行性分析 §17.1](functional-design-and-feasibility.md#171-已确认的最终-ui-设计稿索引)。
- 同一场景的 Superdesign 历史或探索版本不是实现依据。

### 1.2 每个会话的固定产出

1. 先说明将读取的文档、假设和任务边界；发现需求冲突即停下确认。
2. 只修改当前任务包授权的文件；保留并避开无关的工作区改动。
3. 以测试驱动或至少先写可复现用例的方式实现行为和核心规则。
4. 运行与改动相称的单元、集成、类型、格式或端到端检查。
5. 交接时记录：改动、验证命令与结果、未验证项、风险/待决策、下一任务的输入。

## 2. 里程碑与任务包

| 里程碑 | 目标 | 进入条件 | 退出条件 |
| --- | --- | --- | --- |
| M0 架构冻结 | 决定可实现的技术边界与 ADR | UI 已确认 | ADR、数据边界和质量门禁已评审 |
| M1 可运行骨架 | 项目可本地启动并持续验证 | M0 | CI 基线、测试骨架、配置策略可用 |
| M2 冰箱与库存闭环 | 创建冰箱、录入、查看、编辑和删除库存 | M1 | 手机端核心库存路径通过 |
| M3 墨水屏与无账号配对 | Kindle 视图与手机配对可用 | M2 | 端到端配对、权限撤销和低频同步通过 |
| M4 食谱与提醒 | 食谱、补货、临期和设备提醒可用 | M2、M3 | 动态扣库存和提醒规则通过 |
| M5 发布准备 | 风险 spike、兼容性和验收完成 | M3、M4 | 发布清单通过，已知限制明确 |

### P0：架构与 ADR（已完成）

- **目标**：选定前端/PWA、服务端、数据库、认证凭证、文件短效 URL、通知、部署和观测方案。
- **输入**：功能设计 §1–18、最终 UI 索引、目标 Kindle 型号/浏览器范围。
- **输出**：[架构概览](architecture/README.md)、[ADR 索引](architecture/adr/README.md)、数据所有权与信任边界图、关键技术风险清单。
- **完成条件**：能解释无账号配对的凭证生命周期、二维码短效策略、Agnes 图片传递与删除、离线/日同步语义；没有未经确认的技术假设。
- **依赖**：无。

### P1：工程骨架与质量门禁

- **目标**：建立项目结构、环境配置约定、开发/测试/格式化/类型检查命令及 CI 最小闭环。
- **输入**：P0 ADR。
- **输出**：可启动的最小应用、样例环境变量说明、测试框架、CI 配置、项目级 `AGENTS.md`（仅写已验证命令与约定）。
- **完成条件**：新环境按 README 可启动；核心命令在干净工作区通过；机密不进入仓库。
- **依赖**：P0。

### P2：领域模型、迁移与核心规则

- **目标**：实现冰箱、布局、分区、食物分类、库存批次、BBD、设备凭证和食谱的领域模型。
- **输入**：P0 数据模型 ADR、功能设计 §4–9 与 §10。
- **输出**：迁移、类型/schema、仓储/服务边界及规则单元测试。
- **完成条件**：覆盖 BBD 可选、临期窗口 clamp、严格小类匹配、BBD 最早优先扣减、撤销可逆、按大类记忆位置。
- **依赖**：P1。

### P3：无账号配对与设备授权

- **目标**：实现短效二维码会话、PWA 自动配对、独立设备凭证、撤销和重新加入。
- **输入**：P0 安全 ADR、P2 设备与冰箱模型、UI 草稿“配对/冰箱管理/冰箱切换”。
- **输出**：配对 API、凭证存储/轮换策略、访问控制、撤销后的客户端清理路径、端到端测试。
- **完成条件**：非 PWA 不配对；PWA 扫码不需二次确认；移除设备后不能访问；重新扫码可恢复。
- **依赖**：P1、P2。

### P4：冰箱模板、布局配置与位置选择

- **目标**：实现七种模板、图形化分格配置与必填存放位置。
- **输入**：P2、最终 UI 草稿“创建冰箱/布局预览/布局分格编辑/确认位置与数量”。
- **输出**：模板定义、布局编辑 API/界面、位置选择器和持久化测试。
- **完成条件**：单行区域只允许一格、左右两格、左中右三格；其余纵向区域支持 1–6 层；预览与墨水屏布局同构。
- **依赖**：P2。

### P5：库存、分类与图标库

- **目标**：完成库存 CRUD、两级分类、小类图库、用户自定义小类与图标复用。
- **输入**：P2、P4、最终 UI 草稿“添加食材/小类图库/自定义小类/编辑食材”。
- **输出**：库存 API、分类搜索、图标资产接口、编辑/删除流程、相关测试。
- **完成条件**：首页按小类优先、空间不足降级大类；自定义图标确认后可复用；未填 BBD 不参与风险提醒。
- **依赖**：P2、P4。

### P6：相机、条码与 AI 增量识别

- **目标**：实现手机相机、条码识别、Agnes 手工触发识别、字段增量回填与冲突确认。
- **输入**：P0 的 Agnes/媒体安全 ADR、P5、最终 UI 草稿“添加食材/AI 识别确认”。
- **输出**：浏览器能力降级、条码服务适配、短效图片 URL 生命周期、识别适配层、冲突 UI、契约测试。
- **完成条件**：只写入本次识别到的字段；冲突不静默覆盖；二维码/条码与无包装食品均可走各自路径；媒体按 ADR 删除。
- **依赖**：P1、P3、P5。

### P7：手机端日常首页与冰箱管理

- **目标**：实现手机冰箱首页、冰箱切换、设备管理、配对二维码入口及设置页。
- **输入**：P3–P5、最终 UI 草稿“当前冰箱首页/冰箱管理/我的冰箱/提醒设置/临期规则”。
- **输出**：响应式 PWA 页面、访问权限 UI、手工刷新状态、设置持久化、页面测试。
- **完成条件**：当前冰箱切换不会串数据；移除设备立即失效；手机首页与已确认 UI 结构一致。
- **依赖**：P3、P4、P5。

### P8：墨水屏端视图与低频同步

- **目标**：实现竖屏 Kindle 视图、冰箱分区首页、分区详情、配对二维码和离线状态。
- **输入**：P3–P5、最终 UI 草稿“墨水屏端”四项。
- **输出**：墨水屏路由/样式、分区点击、静态翻页、每日同步与手工刷新、离线标记、实机检查清单。
- **完成条件**：无动画/悬停依赖；食品图标不可单点；同层操作符合已确认两行详情；睡眠后唤醒补同步语义正确。
- **依赖**：P3、P4、P5。

### P9：食谱、动态补货与库存扣减

- **目标**：实现本周/下周、文本导入、逐日编辑、动态缺货、复制清单、完成扣减和撤销。
- **输入**：P2、P5、最终 UI 草稿“食谱/导入/编辑/补货”。
- **输出**：解析器、严格匹配服务、补货计算、扣减/撤销事务、手机与墨水屏图标状态、测试。
- **完成条件**：七天固定显示；只精确匹配小类；库存变动即时重算；扣减可逆且不误扣。
- **依赖**：P2、P5、P8（仅墨水屏补货入口）。

### P10：提醒、同步与设备健康

- **目标**：实现每日一次食品提醒、提醒时间/关闭、冰箱端日同步、断网重试和手机端设备健康提醒。
- **输入**：P2、P7、P8、P9。
- **输出**：任务调度、通知适配、同步状态模型、重试策略、模拟时钟测试与观测日志。
- **完成条件**：临期按 `clamp(ceil(总有效期×20%), 1 天, 14 天)`；无 BBD 不提醒；当日未同步仍通知设备异常；冰箱端非睡眠时才每 30 分钟重试。
- **依赖**：P7、P8、P9。

### P11：Spike、端到端验收与发布准备

- **目标**：验证 Kindle、PWA 安装、相机/条码、Agnes、图标生成、食谱解析和安全边界。
- **输入**：P3、P6、P8–P10 与功能设计 §15 的 spike 清单。
- **输出**：测试矩阵、实机截图/记录、性能与失败回退结论、已知限制、发布/回滚清单。
- **完成条件**：高风险场景均有通过证据或已确认的降级方案；关键端到端流程通过；无阻塞级安全问题。
- **依赖**：P3、P6、P8、P9、P10。

## 3. 分会话依赖图

```text
P0 → P1 → P2
          ├→ P3 ─┐
          ├→ P4 → P5 → P6
          │             └→ P7
          │             └→ P8 → P9 → P10
          └───────────────────────┘
P3 + P6 + P8 + P9 + P10 → P11
```

可并行的推荐组合：P3 与 P4；P6 与 P7；P8 与 P9 的服务端部分（但 P9 的墨水屏入口等 P8）。每次并行前先确认任务不会改同一文件或迁移。

## 4. 会话结束的交接格式

每个任务包结束时，将以下内容填入进度看板的会话记录：

```text
任务：P?
状态：完成 / 阻塞 / 进行中
改动：<文件与关键行为>
验证：<命令> — 通过/失败及原因
未验证：<原因与风险>
待决策：<需要产品或架构确认的问题；没有则写“无”>
下一步：<下一个可领取任务及其输入>
```

不要仅以“代码已写完”作为完成标准；必须满足任务包的完成条件并留下验证证据。
