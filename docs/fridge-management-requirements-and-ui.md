# 冰箱资料、已有布局与删除：需求及 UI 设计增补

状态：已确认，作为 P7.1 实施基线  
版本：0.1  
日期：2026-07-24  
关联任务：P7.1  
关联基线：[功能设计与可行性分析](functional-design-and-feasibility.md)、[UI 设计规范](ui-design-specification.md)、[最终 UI 设计图注册表](final-ui-designs.md)

## 1. 背景与问题

当前“我的冰箱”允许创建、打开和切换冰箱，但缺少三个日常管理能力：

1. 创建后不能修改冰箱名称，多个使用默认名称“家里冰箱”的条目无法辨认。
2. 创建页提示“创建后仍可在手机端调整布局”，但已有冰箱没有进入布局编辑器的入口。
3. 所有者不能删除误建、重复或不再使用的冰箱；“设备”页的移除操作只撤销手机或显示设备凭证，不能删除冰箱。

本增补将这三项能力收口到同一个“冰箱设置”入口，并明确所有者权限、缩减分格时的库存自动归位和删除恢复边界。

## 2. 目标与非目标

### 2.1 目标

- 所有者可以重命名已有冰箱，并在所有手机端与冰箱端界面看到一致的新名称。
- 所有者可以从当前持久化布局进入图形化编辑器，修改分格数量和模板允许调整的温区。
- 减少仍有食品的分格时，系统把受影响食品自动归入该区域最后一个保留分格；归位和布局保存原子完成。
- 所有者可以安全删除误建或不再使用的冰箱，并在恢复期内撤销删除。
- “我的冰箱”列表在存在同名历史数据时仍能稳定区分条目。

### 2.2 非目标

- 本任务不允许自由绘制冰箱外形。
- 本任务不支持把已有冰箱从一个模板整体转换成另一个模板。
- 本任务不允许普通设备凭证执行重命名、布局修改、删除或恢复；这些均为所有者专属操作。
- 本任务不恢复删除前的手机或冰箱端设备凭证；恢复冰箱后需要重新配对。
- 本任务不提供跨冰箱迁移库存、食谱或设置，也不提供手动选择库存迁移目标的流程。

## 3. 用户与权限

- **所有者管理会话**：可重命名、编辑布局、删除和恢复自己拥有的冰箱。
- **PWA/浏览器设备凭证**：可继续执行已授权的日常库存和食谱操作，但不能看到破坏性管理入口，也不能调用对应 API。
- **冰箱端显示设备凭证**：只读取自身绑定冰箱并执行既有低频同步/数量操作，不能修改冰箱资料或布局。
- 所有所有者接口对“不存在、已删除、非本人所有”的冰箱统一返回未找到，避免泄漏归属。

## 4. 功能需求

### 4.1 “我的冰箱”与管理入口

- 每个活跃冰箱条目显示名称、模板摘要、库存数量和当前/非当前状态；即使名称相同也能区分。
- 每个条目保留独立的 48px“打开冰箱”图标按钮，进入该冰箱原有的当前冰箱首页；右侧独立的 48px“冰箱设置”图标直接进入该条目的设置页，不能把打开和管理混成一个点击目标。
- “编辑名称”“编辑布局”“设备访问”“删除冰箱”只在冰箱设置页出现；列表不提供重复的弹出管理菜单。
- 当前冰箱首页的“管理冰箱”入口进入同一“冰箱设置”页，不能另建一套行为不同的入口。
- 列表底部在存在软删除记录时显示“最近删除”分组与可恢复条目；恢复成功后条目立即从该分组移除。

### 4.2 编辑冰箱名称

- 名称去除首尾空白后长度为 1–120 个字符；空名称不能保存。
- **推荐规则（待评审）**：同一所有者下，去除首尾空白后的名称必须唯一。历史同名数据不自动改名，但新建或重命名不能继续制造同名条目。
- 重命名成功后，列表、当前冰箱首页、设备管理、提醒设置和冰箱端显示在下一次读取/同步时使用新名称。
- 保存期间禁用重复提交；失败时保留输入并显示可继续操作的错误信息。
- 名称未发生变化时返回上一页，不发送写请求。
- 并发删除或权限变化导致保存失败时，退出编辑页并刷新冰箱列表。

### 4.3 编辑已有布局

- 从“冰箱设置”进入布局预览，复用已冻结的“冰箱布局预览”和“布局分格编辑”视觉结构，但以当前持久化布局作为初始值。
- 外形模板及其大区集合保持不变；本任务只允许：
  - 纵向区域在 1–6 层之间调整；
  - 单行区域在一格、左右两格、左中右三格之间调整；
  - 模板声明为可调温的区域在冷藏与冷冻之间切换。
- 增加分格可以直接保存。
- 减少空分格可以直接保存。
- 减少有食品的分格时，系统把所有将移除分格内的库存批次自动归入该区域按物理顺序最后一个保留分格。例如冷藏室从五层改为三层时，原第四、五层的食材都归入第三层。
- 布局变更、库存自动归位和失效位置记忆清理必须在同一事务中完成；任一失败不得留下半保存状态。
- 打开编辑器后若库存或布局已被其他设备修改，保存时返回冲突；客户端重新读取最新数据，不静默覆盖。
- 保存成功后刷新手机端位置选择器和冰箱预览；冰箱端显示设备按既有同步规则读取新布局。

### 4.4 删除、恢复与永久清理

- 删除入口仅出现在“冰箱设置”的危险操作区；列表只提供直达设置页的入口。
- 删除确认页显示冰箱名称、库存种类/数量、已绑定设备数量以及删除影响。
- 用户必须输入当前冰箱名称才能启用“删除冰箱”；即使历史数据存在同名冰箱，服务端仍按稳定冰箱 ID 删除当前上下文中的条目。
- 删除采用软删除：
  - 冰箱立即从活跃列表隐藏；
  - 所有 PWA、浏览器和冰箱端设备凭证立即撤销；
  - 当前冰箱被删除时清空客户端工作区并返回“我的冰箱”；有其他冰箱时不自动猜测切换目标；
  - 库存、布局、食谱和设置在恢复期内保持原样，但不能通过活跃 API 访问。
- 软删除记录保留 30 天。所有者可在“最近删除”中恢复，逾期后由单实例调度任务永久清理；该规则沿用已接受的架构与调度边界。
- 恢复后冰箱重新出现在列表中；原设备凭证保持撤销，必须重新配对。
- 永久清理删除冰箱及其布局、库存、分类扩展、位置记忆、食谱、提醒设置、设备凭证和业务审计关联；备份仍按既有运维保留策略处理。
- 允许删除最后一台冰箱；删除后进入空列表/新建冰箱状态。

## 5. UI 信息架构

```text
我的冰箱
├── 打开图标 → 该冰箱原有的当前冰箱首页
├── 冰箱设置 → 编辑名称 / 编辑布局 / 设备访问（已绑定或未绑定）/ 删除冰箱
├── 新建冰箱
└── 最近删除 → 恢复冰箱

当前冰箱首页
└── 管理冰箱 → 冰箱设置
    ├── 编辑名称
    ├── 编辑布局
    ├── 设备访问管理
    └── 危险操作：删除冰箱
```

## 6. 页面设计

所有页面遵循 [UI 设计规范](ui-design-specification.md)：390×844px 为手机基准，二级页使用三列 `PageHeader`，交互热区至少 48×48px，危险操作不能与主操作同色同权重。

### 6.1 我的冰箱

- 顶部为二级页面导航栏：返回、居中标题“我的冰箱”、右侧“新建冰箱”图标。
- 列表条目依序显示冰箱轮廓、名称/模板/库存摘要、48px“打开冰箱”图标和右侧独立 48px“冰箱设置”图标；两个图标同尺寸、同权重，均带文字辅助标签；打开图标进入原有当前冰箱首页，齿轮进入设置页，不弹出二级菜单。
- 当前条目使用粗边框和文字“当前冰箱”，不能只依赖颜色。
- “最近删除”位于列表末尾分组，不与活跃冰箱混排。

### 6.2 冰箱设置

- 顶部标题为“冰箱设置”，内容区首先展示冰箱轮廓和当前名称。
- 身份区提供“编辑名称”和“编辑布局”两个等权次级操作。
- 设备访问合并原“冰箱管理与设备访问”页：保留可访问手机的列表与既有移除能力。
- “冰箱端”按绑定状态展示：已绑定时显示终端名称和同步状态；未绑定时显示“尚未连接冰箱端设备”，并提供“生成兼容绑定码”次级操作。绑定码沿用既有配对规则，为六位数字、五分钟有效，仅所有者可生成。
- 页面末尾增加“危险操作”分组；“删除冰箱”使用红色文字、描边和明确说明，不使用主按钮样式。

### 6.3 编辑冰箱名称

- 独立二级页，顶部返回、居中标题、右侧“保存”。
- 单一输入框预填当前名称，自动聚焦但不自动全选。
- 发生同名冲突时在输入框下显示“已有同名冰箱，请换一个名称”，保存按钮保持可用以便用户修正后重试。
- 保存成功返回冰箱设置页，并用非阻塞状态提示确认。

### 6.4 编辑布局与自动归位

- 预览和布局编辑复用草稿 `e5c35dea-610f-42f9-878b-1f716c2e7d4f` 与 `145b32f6-007a-4698-9ea7-3963dfc04a38`。
- 实现复用现有 P4 的 `OpenFridge` 图形预览、分区标签、分格选项和温区选择面板；不新建第二套布局编辑页面。已有冰箱只是在进入编辑器时以已持久化布局初始化，保存后返回冰箱设置页而非新建流程预览。
- 已有布局预览的主操作文案为“保存布局”，次操作为“编辑布局”；不能显示“创建冰箱”。
- 减少分格时，在编辑页就近显示说明：将被删除分格中的食材会自动归入该区域最后一个保留分格；展示受影响食品摘要以便用户知情。
- 页面底部固定主操作“保存布局”；不出现“迁移食材”页或目标位置选择器。返回时保留尚未提交的布局草稿。

### 6.5 删除冰箱

- 使用独立确认页，不使用浏览器原生 `confirm()`。
- 顶部标题“删除冰箱”，正文首先说明立即影响，再说明 30 天恢复期。
- 展示冰箱名称、库存和设备摘要；要求输入名称。
- 主操作区只有危险按钮“删除冰箱”，未匹配时禁用；次操作“取消”返回冰箱设置。
- 删除成功后不显示成功弹层，直接回到“我的冰箱”并显示“已移到最近删除，可在 30 天内恢复”。

### 6.6 最近删除

- 列表显示名称和恢复提示；恢复成功后直接从最近删除分组移除。
- “恢复”需要一次轻量确认；恢复成功返回活跃列表。
- 不提供提前永久删除入口，避免把恢复与不可逆清理放在同一首版流程中。

## 7. 状态与错误

| 场景 | UI 行为 |
| --- | --- |
| 加载 | 保留页面壳和标题，内容显示骨架或短文案 |
| 空列表 | 说明尚无冰箱并提供“新建冰箱”主操作 |
| 历史同名 | 卡片显示模板/库存摘要；冰箱设置按稳定 ID 作用于当前卡片 |
| 名称冲突 | 输入框下显示冲突，不丢失输入 |
| 布局保存冲突 | 提示数据已变化，重新载入最新布局和库存 |
| 自动归位失败 | 取消整个保存并保持原布局和库存，提示重试 |
| 删除失败 | 保留确认页和输入，显示失败原因 |
| 当前冰箱被其他会话删除 | 清空当前工作区，返回列表并提示访问已移除 |
| 最近删除为空 | 显示“没有可恢复的冰箱”，保留返回入口 |
| 离线 | 管理写操作禁用并提示联网后重试；不做本地乐观删除 |

## 8. API 与数据契约草案

具体 schema 名称可按项目现有约定调整，但行为不得改变：

```text
PUT    /api/owner/refrigerators/{id}
       { name }

PUT    /api/owner/refrigerators/{id}/layout
       { expected_revision, zones }

DELETE /api/owner/refrigerators/{id}
       { confirmation_name }

GET    /api/owner/refrigerators/deleted

POST   /api/owner/refrigerators/{id}/restore
```

- 冰箱和布局需要可比较的修订标识或 `updated_at`，用于拒绝过期写入。
- 布局服务根据旧、新布局计算被移除的分格；有库存时自动将其归入相同区域最后一个保留分格，并与布局替换在同一事务提交。
- 删除与恢复必须在服务层集中处理关联数据和设备凭证，不能由路由逐表拼装。
- 列表 API 默认只返回活跃冰箱；已删除列表使用独立所有者接口。
- 所有 Python 新增公共服务方法必须有 Google 风格 docstring。

## 9. 待评审决策

以下两项已于 2026-07-24 产品确认：

1. **名称唯一性**：禁止继续创建或重命名为同名冰箱；历史重复数据不自动改写。
2. **已有布局范围**：首版保持外形模板和大区集合不变，只编辑分格与可调温区；整套模板迁移另立需求。

删除采用 30 天软删除、恢复不恢复设备凭证并到期永久清理，已经由 [架构概览](architecture/README.md) 和 ADR 确认，不作为本次重新决策项；本次只评审对应 UI 流程和文案。
